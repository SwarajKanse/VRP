# Requirements Document: Gravity & Density Upgrade

## Introduction

This feature upgrades the VRP solver's packing engine with a **Gravity-Driven Deepest-Bottom-Left (DBL)** algorithm. It introduces strict physics constraints (gravity and support checks) and density optimization while **strictly preserving** the LIFO (Last-In-First-Out / Reverse Stop Order) logic required for delivery operations.

The DBL approach ensures packages are stacked densely, strictly adhere to gravity constraints (no floating boxes), and form stable load configurations for safe transport. This upgrade addresses real-world logistics concerns including load stability, space utilization, and safe handling during delivery operations, while maintaining the critical LIFO constraint that ensures the last delivery stop is loaded first (and thus accessible first during unloading).

## Glossary

- **Packing_Engine**: The component responsible for determining 3D placement positions for packages within a vehicle's cargo space
- **DBL (Deepest-Bottom-Left)**: A packing heuristic that minimizes the Euclidean Distance to the back-bottom-left corner (0,0,0), prioritizing dense packing toward the origin
- **Contact_Point**: A candidate 3D coordinate (x, y, z) where a new package can potentially be placed, generated by projecting lines from existing package corners
- **Support_Surface**: The top surface area of a package that can support other packages placed above it
- **Support_Ratio**: The percentage of a package's bottom surface area that is supported by packages below it (must be ≥ 80%)
- **LIFO_Constraint**: The mandatory sorting rule where packages are loaded in reverse stop order (last delivery loaded first) to enable efficient unloading
- **Stop_Group**: A collection of packages destined for the same delivery stop
- **Euclidean_Distance**: The straight-line distance from a point to the origin (0,0,0) in 3D space, calculated as √(x² + y² + z²)
- **Cargo_Space**: The 3D rectangular volume inside a vehicle available for package placement
- **Package**: A 3D rectangular box with dimensions (length, width, height), weight, and destination stop ID
- **Bridge_Support**: A placement where a package sits across multiple packages below, with gaps between them, as long as total support ≥ 80%

## Requirements

### Requirement 1: Strict Gravity and Support Constraint

**User Story:** As a driver, I want packages to be stacked with proper support, so that the load remains stable and doesn't collapse during transport.

#### Acceptance Criteria

1. WHEN placing a package at height z > 0, THE Packing_Engine SHALL verify that at least 80% of the package's bottom surface area is supported by the top surfaces of packages below it
2. IF the Support_Ratio is less than 80%, THEN THE Packing_Engine SHALL reject the placement as invalid
3. THE Packing_Engine SHALL allow packages at z = 0 (floor level) without support validation
4. WHEN calculating Support_Ratio, THE Packing_Engine SHALL compute the intersection area between the package's bottom surface and all supporting package top surfaces using rectangle intersection logic
5. WHEN a package sits across multiple packages (Bridge_Support), THE Packing_Engine SHALL sum the overlapping areas to calculate total support, allowing gaps as long as total support ≥ 80%

### Requirement 2: Deepest-Bottom-Left Placement Heuristic

**User Story:** As a fleet manager, I want to maximize cargo space utilization, so that I can reduce the number of vehicles needed and lower operational costs.

#### Acceptance Criteria

1. WHEN placing a new package, THE Packing_Engine SHALL generate a list of Contact_Points from currently placed packages
2. THE Packing_Engine SHALL attempt to place the package at each Contact_Point in the candidate list
3. WHEN evaluating placement positions, THE Packing_Engine SHALL calculate the Euclidean_Distance from each valid position to the origin (0,0,0) using the formula √(x² + y² + z²)
4. THE Packing_Engine SHALL select the placement position with the minimum Euclidean_Distance (closest to back-bottom-left corner)
5. WHEN Euclidean distances are equal, THE Packing_Engine SHALL use tie-breaker rules: prioritize lowest Z coordinate, then lowest X coordinate, then lowest Y coordinate
6. IF no valid Contact_Point exists after evaluating all candidates, THEN THE Packing_Engine SHALL report that the package cannot be placed

### Requirement 3: LIFO Sorting with Weight-Based Secondary Sort

**User Story:** As a loader, I need to load packages in reverse stop order (LIFO) for efficient unloading, but within each stop I want heavier items at the bottom to prevent crushing.

#### Acceptance Criteria

1. THE Packing_Engine SHALL sort all packages by reverse stop order as the primary sort criterion (Stop 5 → Stop 4 → ... → Stop 1)
2. WITHIN each Stop_Group, THE Packing_Engine SHALL sort packages by weight in descending order (heaviest first) as the secondary sort criterion
3. WHEN placing Package_A on top of Package_B, THE Packing_Engine SHALL verify that Weight(Package_A) ≤ Weight(Package_B) × 1.5
4. IF the weight constraint is violated, THEN THE Packing_Engine SHALL attempt to place the package at a different Contact_Point (e.g., on the floor or on a different supporting package)
5. WHEN a package is supported by multiple packages below (Bridge_Support), THE Packing_Engine SHALL verify the weight constraint against each supporting package

### Requirement 4: Contact Point Generation

**User Story:** As a system developer, I want an efficient method to identify valid placement positions, so that the packing algorithm runs quickly even with many packages.

#### Acceptance Criteria

1. WHEN the cargo space is empty, THE Packing_Engine SHALL initialize the Contact_Point list with the origin (0, 0, 0)
2. WHEN a new package P is placed, THE Packing_Engine SHALL generate new Contact_Points at the following positions:
   - (P.x + P.length, P.y, P.z) - right edge
   - (P.x, P.y + P.width, P.z) - front edge
   - (P.x, P.y, P.z + P.height) - top corner
3. THE Packing_Engine SHALL filter out Contact_Points that are outside the vehicle boundaries
4. THE Packing_Engine SHALL filter out Contact_Points that are already occupied by another package
5. THE Packing_Engine SHALL maintain an updated list of valid Contact_Points after each package placement

### Requirement 5: Placement Validation and Error Reporting

**User Story:** As a quality assurance engineer, I want comprehensive validation of package placements with clear error reporting, so that I can diagnose why packages fail to fit.

#### Acceptance Criteria

1. WHEN validating a placement, THE Packing_Engine SHALL check constraints in the following order: boundary check, overlap check, support check, stackability check
2. IF any constraint is violated, THEN THE Packing_Engine SHALL reject the placement and try the next Contact_Point
3. WHEN a package cannot be placed at any Contact_Point, THE Packing_Engine SHALL report which constraint(s) caused the failure
4. THE Packing_Engine SHALL distinguish between "Overflow due to Size" (package too large) and "Overflow due to Stability" (failed support or weight constraints)
5. THE Packing_Engine SHALL use rectangle intersection logic (not pixel/grid counting) to calculate support areas efficiently

### Requirement 6: Integration with Existing VRP Solver

**User Story:** As a system architect, I want the DBL packing engine to integrate seamlessly with the existing VRP solver, so that routing and packing work together without disruption.

#### Acceptance Criteria

1. THE Packing_Engine SHALL accept the same input format as the existing LIFO packing engine (package list with dimensions, weight, and stop ID)
2. THE Packing_Engine SHALL return placement results in the same format as the existing engine (package ID, position, orientation)
3. WHEN the DBL algorithm is enabled, THE Packing_Engine SHALL replace the First-Fit logic while maintaining the same API
4. THE Packing_Engine SHALL support configuration to switch between LIFO and DBL algorithms
5. THE Packing_Engine SHALL maintain compatibility with the Python dashboard visualization

### Requirement 7: Performance Requirements

**User Story:** As a logistics operator, I want the packing algorithm to run quickly, so that I can generate routes and load plans in real-time during dispatch operations.

#### Acceptance Criteria

1. WHEN packing N packages, THE Packing_Engine SHALL complete in O(N²) time complexity or better
2. THE Packing_Engine SHALL generate and evaluate Contact_Points efficiently without exhaustive grid search
3. WHEN placing a package, THE Packing_Engine SHALL evaluate at most O(N) Contact_Points where N is the number of already-placed packages
4. THE Packing_Engine SHALL cache computed values (support areas, distances) to avoid redundant calculations
5. THE Packing_Engine SHALL complete packing for a typical delivery route (20-50 packages) in under 1 second on standard hardware
